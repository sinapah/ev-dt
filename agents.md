# EV Charging Smart City Simulation: Federated Learning + Digital Twin + Demand Routing

## Objective

Implement a simulation framework for a smart-city EV charging management system.

The system consists of:

1. Two physical charging edge nodes:

   * Caltech charging park
   * JPL charging park

2. One centralized Digital Twin (DT) representing both charging parks.

3. One centralized scheduler responsible for routing charging demand between the two parks.

The objective is to evaluate whether a Digital Twin can improve charging system efficiency by predicting future congestion and advising a scheduler on how to distribute incoming EV charging requests.

The simulation must be designed so that a future phase can introduce telemetry outages where the Digital Twin must operate using synthetic telemetry generated by KDE/WGAN models. Do not implement outages yet.

---

# Dataset Description

The input file is:

```
ground_truth.csv
```

This file represents the ground truth behavior of the physical charging system.

The file was created by:

1. Taking the original ACN charging datasets:

   * Caltech
   * JPL
   * Office1

2. Replaying each charging site using a discrete-event charging simulator.

3. Creating queues by limiting the number of available chargers.

4. Aggregating the simulated results into hourly telemetry.

5. Filtering the resulting dataset to only include timestamps where BOTH Caltech and JPL have available records.

Therefore:

* Each row represents one hour.
* The dataset represents the real observed behavior of the physical system.
* The values in this file are the ground truth and should not be generated synthetically.

The dataset contains fields similar to:

```
timestamp_hour

Caltech:
    arrivals_per_hour
    average_service_time_minutes
    average_waiting_time_minutes
    average_queue_length
    maximum_queue_length
    charger_utilization
    active_charging_sessions
    completed_sessions

JPL:
    arrivals_per_hour
    average_service_time_minutes
    average_waiting_time_minutes
    average_queue_length
    maximum_queue_length
    charger_utilization
    active_charging_sessions
    completed_sessions
```

# Simulation Time Model

The simulation progresses hourly.

Each timestep corresponds to one row in:

```
ground_truth.csv
```

The simulation should iterate chronologically through the timestamps.

At timestep t:

The environment provides the actual physical state:

```
Caltech(t)

JPL(t)
```

The environment represents the real charging system.

The environment should not make decisions.

# System Components

Implement the following classes:

```
Environment
EdgeNode
FederatedCoordinator
DigitalTwin
Scheduler
SimulationRunner
```

The code should be modular and object-oriented.

# 1. Environment

The Environment class represents the real charging infrastructure.

Responsibilities:

* Load ground_truth.csv.
* Iterate through timestamps.
* Provide the current ground truth state.

At each timestep, the environment should expose:

For Caltech:

```
actual_arrivals
actual_queue_length
actual_waiting_time
actual_utilization
```

For JPL:

```
actual_arrivals
actual_queue_length
actual_waiting_time
actual_utilization
```

The environment also provides the actual incoming demand that must be routed.

The total incoming demand at timestep t is:

```
actual Caltech arrivals + actual JPL arrivals
```

This represents the real charging demand arriving to the city.

---

# 2. Physical Edge Nodes

There are two physical edge nodes:

```
EdgeNode_Caltech

EdgeNode_JPL
```

Each edge node represents the computing infrastructure located at a charging park.

Each edge node only has access to its own local telemetry.

For example:

Caltech edge receives:

```
historical Caltech arrivals
historical Caltech queue length
historical Caltech utilization
historical waiting times
time features
```

JPL receives equivalent local information.

The edge nodes must not access the other site's raw data.

---

# Edge Node Prediction Task

The edge nodes perform demand forecasting.

The prediction target is:

```
next_hour_arrivals
```

Example:

At time t:

Caltech edge predicts:

```
predicted arrivals at t+1
```

JPL edge predicts:

```
predicted arrivals at t+1
```

Use a simple forecasting model initially.

Acceptable models:

* Linear Regression
* Random Forest
* Gradient Boosting
* LSTM

The implementation should allow replacing the model later.

---

# 3. Federated Learning

Implement a simple federated learning workflow.

The purpose is to allow both charging parks to learn a shared demand prediction model without sharing raw telemetry.

Workflow:

1. Each edge node trains locally using only its own data.

2. Local model parameters are sent to:

```
FederatedCoordinator
```

3. The coordinator performs FedAvg.

4. The updated global model is returned to both edge nodes.

The goal is not state-of-the-art forecasting.

The goal is to demonstrate distributed learning between charging sites.

---

# 4. Digital Twin

Implement one centralized Digital Twin.

The Digital Twin represents the complete charging network:

```
Virtual Caltech charging park

Virtual JPL charging park
```

The DT does not train prediction models.

It receives information from the physical edge nodes.

---

## Digital Twin Inputs

The DT receives:

Current physical state:

```
current queue length
current utilization
current waiting time
```

Predicted future demand:

```
Caltech predicted arrivals
JPL predicted arrivals
```

---

## Digital Twin Responsibilities

The DT performs predictive analysis.

Based on predicted demand, estimate future operating conditions:

For each charging park:

```
predicted queue length

predicted waiting time

predicted utilization

predicted congestion level
```

The initial implementation may use:

* simple queue calculations
* heuristic prediction
* lightweight simulation

The design should allow adding a G/G/s queueing model later.

---

# 5. Scheduler

The scheduler controls charging demand allocation.

The scheduler does NOT generate demand.

Demand comes from the ground truth environment.

The scheduler decides where incoming EV requests should be routed.

---

## Scheduler Inputs

The scheduler receives:

1. Actual incoming demand:

Example:

```
Total EV requests:
30
```

2. Digital Twin recommendations:

Example:

```
Caltech:
predicted waiting time = 20 minutes

JPL:
predicted waiting time = 5 minutes
```

---

## Scheduler Objective

The scheduler should route demand to minimize:

* queue length
* waiting time
* congestion
* load imbalance

Example:

Without DT:

```
Caltech:
25 EVs

JPL:
5 EVs
```

DT recommendation:

```
Caltech is predicted to become congested.
```

Scheduler decision:

```
Caltech:
18 EVs

JPL:
12 EVs
```

```

```

```

DT recommendations
```

---

Store all simulation outputs.
